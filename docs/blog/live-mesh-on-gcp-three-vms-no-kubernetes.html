Live Mesh on GCP: Three VMs, No Kubernetes

We deployed a production-grade Skr8tr mesh on Google Cloud Platform — a conductor, two worker nodes, post-quantum authentication, and a live status page served by the orchestrator itself. No Kubernetes. No Docker. No YAML. No control plane overhead.

What We Built

The mesh runs on three GCP VMs in us-central1-a on a dedicated VPC (10.10.0.0/24): 

skr8tr-conductor   e2-small   10.10.0.10   skr8tr_sched + skr8tr_reg
skr8tr-node-1      n2-std-2   10.10.0.11   skr8tr_node
skr8tr-node-2      n2-std-2   10.10.0.12   skr8tr_node

The conductor runs the scheduler (skr8tr_sched) and service registry (skr8tr_reg). The two worker nodes each run skr8tr_node, which connects to the conductor over UDP, sends heartbeats with CPU and RAM availability, and receives workload dispatch commands.

There is no master. The conductor is a scheduler process — it can run on any node. If it restarts, the worker nodes re-register automatically. If a worker dies, the conductor reschedules its workloads. No etcd. No leader election. No quorum.

The Live Status Page

The demo page running at the time of this post is itself a Skr8tr workload — a small C HTTP daemon (skr8tr_demo, ~300 lines) that on each request queries the conductor for live mesh state and returns a styled HTML page. It refreshes every 5 seconds.

Live mesh status — conductor responding, two nodes registered, three workloads running. The page you're looking at is one of those workloads.

Deployed with a single manifest: 

app skr8tr-demo
  exec /usr/local/bin/skr8tr_demo
  args --host 10.10.0.10 --port 8080
  port 8080
  replicas 1

One command. The conductor picked the least-loaded node, sent the dispatch over UDP, the node launched the process, and the demo was live within a second.

Post-Quantum Auth in Practice

Every skr8tr up, skr8tr down, and skr8tr list command is signed with an ML-DSA-65 private key before it reaches the conductor. The conductor verifies the signature against the registered public key before executing anything. Unsigned commands are rejected with UNAUTHORIZED. 

# Generate a keypair (ML-DSA-65, NIST FIPS 204)
skrtrkey keygen
# → skrtrview.pub   (1952 bytes, deploy alongside conductor)
# → ~/.skr8tr/signing.sec   (4032 bytes, operator keeps this)

# Every command auto-signs with your key
skr8tr up app.skr8tr

There are no passwords. No TLS certificate rotation schedules. No PKI to maintain. The keypair is generated once and the identity is permanent until you rotate it.

One Discovery Problem We Fixed

GCP VPC networks do not forward UDP broadcast packets — 255.255.255.255 is dropped at the hypervisor layer. Skr8tr's original heartbeat implementation used INADDR_BROADCAST for node discovery, which works on physical LANs but silently fails on every major cloud VPC.

The fix is clean: when --conductor <ip> is specified, heartbeats are sent as unicast directly to that IP instead of broadcast. Broadcast is preserved as the fallback for local network deployments where the conductor address isn't known in advance. 

# On-prem / LAN — conductor IP not needed, broadcast discovery works
skr8tr_node &

# Cloud / VPC — specify conductor IP for unicast heartbeats
skr8tr_node --conductor 10.10.0.10 &

One flag. Cloud-native without any architectural compromise.

No Docker, No OCI — Bare Processes

Every workload on this mesh is a native binary running directly on the host. No container runtime. No image layers. No overlay filesystem. When the conductor dispatches a workload, the node does the equivalent of: 

execv("/usr/local/bin/skr8tr_demo", argv);

The process starts. It runs. Skr8tr tracks its PID, health-checks it if configured, and restarts it on failure. That's the entire runtime model. No daemon overhead, no cgroup setup, no network namespace creation. Cold deploy time under one second.
